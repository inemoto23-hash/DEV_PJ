# ドキュメント階層ルール

`90_DOCUMENT/` 配下のドキュメントは、以下の3階層で役割を厳密に分離する。
情報の重複を避け、各情報の **単一情報源（SSOT: Single Source of Truth）** を明確化することで、ドキュメント間の不整合を防ぐ。

---

## 階層定義

| 階層 | ドキュメント | 役割 | 更新頻度 |
|------|------------|------|---------|
| **L1: 鳥瞰** | `system_architecture.md`<br>`system_flow.md` | 全体像・コンポーネント間の関係性のみを記載。詳細仕様は書かない。 | 構造変更時のみ |
| **L2: 詳細** | `01_開発ドキュメント/*.md` | ソースファイル単位の責務・I/F・依存関係を記述。 | 該当ファイル変更時 |
| **L3: 定義** | `10_DB/`<br>`pubspec.yaml` 等 | テーブル定義・依存ライブラリ等の機械可読な事実情報。 | 該当変更時 |

---

## 記載ルール

### L1（鳥瞰）に書くこと
- コンポーネント名と役割の **一行サマリ**
- コンポーネント間の関係（依存・呼び出し・データフロー）
- 外部連携先の一覧
- 詳細は L2 へのリンクで委譲する

### L1 に書かないこと
- 関数の引数・戻り値などの I/F 詳細
- ファイル内部のロジック説明
- L2・L3 に記載すべき情報の重複

### L2（詳細）に書くこと
- ソースファイル単位の責務
- 公開 API の I/F 仕様
- 他ファイル・他コンポーネントへの依存
- 内部状態・副作用の有無

### L3（定義）に書くこと
- DB スキーマ
- ストアドプロシージャ・SQL クエリ
- 依存ライブラリのバージョン・用途

---

## SSOT 優先順位

**情報が複数階層で重複した場合、下位階層（より具体的な階層）を正とする。**

- L1 と L2 で内容が異なる → **L2 を正** とし、L1 を更新する
- L2 と L3 で内容が異なる → **L3 を正** とし、L2 を更新する
- L2 とソースコード本体で内容が異なる → **ソースコードを正** とし、L2 を更新する

---

## 更新時の判断基準

変更を行う際、以下の順で「どの階層に影響するか」を判定する。

1. **構造変更（コンポーネント追加・削除・関係変更）か？** → L1 更新必須
2. **公開 I/F や責務の変更か？** → L2 更新必須
3. **DB・依存ライブラリの変更か？** → L3 更新必須

複数階層に影響する場合は、**下位階層から先に更新**し、最後に L1 を整合させる。
