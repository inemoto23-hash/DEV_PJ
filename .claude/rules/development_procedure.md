# 開発手順ルール

ユーザーからの開発指示やアップデート指示があった場合、下記フローを遵守すること。

## 1. システム全体の把握
`90_DOCUMENT/system_architecture.md` および `system_flow.md` を確認し、全体像を把握する。

## 2. ドキュメント検索
`90_DOCUMENT/` 配下をサブエージェント（Explore エージェント）を並列で走らせて検索する。

**ドキュメント検索スキル**
- `subagent_type: Explore` を使用
- 検索対象: `90_DOCUMENT/**/*.md`
- 目的: 関連する既存ドキュメント・仕様・実装情報の収集

## 3. 外部情報取り込み
対象機能・ライブラリの公式ドキュメントを確認し、1・2 の情報と合わせて実装方針を策定する。
- Flutter 公式: https://docs.flutter.dev
- Android Developers: https://developer.android.com/docs

## 4. Q&A
不足している情報・曖昧な仕様に対してユーザーへ質問を行う。
実装を進める前に認識齟齬をなくすこと。

## 5. 計画
1〜4 の情報を元に **Plan モード** を使用し、実装計画をユーザーに提示する。
- 計画書の保存先: `.claude/plans/YYYYMMDD_<タイトル>.md`

### 5-α. 軽微修正の場合（計画スキップ可）
以下のいずれにも該当する変更は **計画フェーズをスキップ** し、`6. 実装` に直接進んでよい。

**スキップ可能な条件（すべて満たすこと）**
- 変更対象が単一ファイル、または密結合した数行の修正
- 公開 API・データモデル・外部連携・画面フローに影響しない
- 新規依存ライブラリの追加を伴わない
- ユーザー手動手順（`40_実行手順/`）の追加が不要

**該当例**
- typo・文言修正
- ログ出力の追加・調整
- 既存関数内のロジック微調整、リファクタ
- コメント追加、フォーマット修正

**該当しない例（計画必須）**
- 新規画面・新規機能の追加
- DB スキーマ・データモデルの変更
- 外部 API 連携の追加・変更
- 複数ファイルにまたがる構造変更

判断に迷う場合は **計画を作成する側に倒す**こと。スキップ時は `6. 実装` 冒頭で「軽微修正につき計画スキップ」と明示してから着手する。

## 6. 実装
計画が承認された後（軽微修正の場合は 5-α の判断後）、以下の順で進める。

1. `91_UPDATE/update/YYYYMMDD_update.md` へアップデート予定内容を記載する（軽微修正時は計画書パスの代わりに「軽微修正」と明記）
2. ユーザーが手動で実施すべき手順がある場合、**実装前に** `90_DOCUMENT/40_実行手順/YYYYMMDD_実行手順.md` へ出力する
3. 実装を進める

## 7. ドキュメント更新
実装完了後、必ず以下を更新する：
- `90_DOCUMENT/system_architecture.md` — 今回の変更を反映
- `90_DOCUMENT/system_flow.md` — フロー変更があれば更新
- `91_UPDATE/update_fix/YYYYMMDD_updatefix.md` — 完了したアップデート内容を記述

更新の際は **`.claude/rules/document_hierarchy.md`（ドキュメント階層ルール）** に従い、各階層の役割分担と SSOT 優先順位を遵守すること。
- L1（鳥瞰）には詳細を書かず、L2 へのリンクで委譲する
- 複数階層に影響する変更は、下位階層から先に更新する
